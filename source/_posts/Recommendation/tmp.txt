## 大规模数据处理中的数据倾斜治理
数据倾斜的本质是 "热点 Key" 导致分布式计算中个别 Task 处理的数据量远超其他 Task，从而拖慢整个 Job 的进度（长尾效应）。

场景一：聚合操作（GroupBy/Count）引起的倾斜
现象：热门商品（爆款 ID）的点击量极高，导致 reduce 阶段某个分区内存溢出（OOM）或运行极慢。

核心对策：两阶段聚合（加盐局部聚合 + 全局聚合）
第一阶段（局部聚合）：给 Key 加上随机前缀（如 0~9_Item_A），将原本路由到同一个分区的热点 Key 分散到多个分区进行预聚合。

第二阶段（最终聚合）：去掉随机前缀，对预聚合的结果进行二次汇总。

适用场景：count, sum 等满足结合律的聚合运算。

面试加分：在 Spark 中，可以提到 spark.sql.adaptive.enabled（自适应查询执行 AQE），它能自动检测并处理倾斜。

场景二：连接操作（Join）或序列提取引起的倾斜
现象：按 User_ID 分组提取行为序列时，爬虫或超级活跃用户产生的行为日志成千上万，导致 Shuffle 过程极其缓慢。

核心对策：过滤异常 + 广播 Join + 随机采样
过滤与分流：

爬虫治理：识别出行为频次远超常人的 ID，单独存储或在特征计算中剔除。

分而治之：将倾斜的 Key 拆分出来单独处理，不倾斜的按常规逻辑。

Broadcast Join（大表 Join 小表）：

如果不涉及分组序列，只是为了关联属性，使用 广播变量 将小表分发到每个 Executor，避免 Shuffle。

原理：将 Reduce-side Join 转化为 Map-side Join。

采样与扩容：

对大表进行采样，找出 Top N 个导致倾斜的 Key，对这些 Key 对应的另一张表的数据进行“膨胀”（拷贝多份），从而配合大表加盐后的并行计算。

场景三：特征编码与类别不平衡引起的倾斜
现象：地理位置中的“未知”或兴趣标签中的“其他”占比过高，导致特征工程（如 One-Hot 编码或频率统计）时压力集中。

核心对策：特征平滑与重分区
类别重组：将极低频的类别归为 Other，对极高频的默认值（如 Unknown）进行特殊标记或随机映射。

重分区 (Re-partition)：

在处理特征前手动调用 .repartition(num_partitions, "feature_col")。

深度理解：Spark 默认的 HashPartitioner 可能导致数据分布不均，可以自定义 RangePartitioner 或增加分区数来缓解。

💡 面试官深度追问 (Deep Dive)
Q1：如何定位是哪个 Key 导致了倾斜？

回答：

查看 Spark UI：观察各个 Task 的 Shuffle Read Size/Records，如果某个 Task 的数据量远大于 Median，即存在倾斜。

日志采样：对原始数据进行采样 df.sample(...).groupBy("key").count().sort(desc)，找出频率异常高的 Key。

Q2：如果不能加盐（比如必须按 ID 严格排序提取序列），怎么办？

回答：

窗口函数优化：限制每个用户的序列长度（截断），只保留最近的 N 条。

资源倾斜调整：增加 Executor 的内存或调大 spark.sql.shuffle.partitions。

Q3：Flink 实时流处理中如何解决数据倾斜？

回答：

使用 Window 时的增量聚合（AggregateFunction）。

利用 Flink 的 MiniBatch 机制，减少状态访问频率。

对于 KeyBy 倾斜，同样可以使用“二次聚合”的思想。
